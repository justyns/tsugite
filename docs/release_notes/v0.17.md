# v0.17

## 0.17.3 (2026-07-09)

### Features

- Jobs can retry on a different model: the retry dialog and `POST /api/jobs/{id}/retry` take an optional `model`/`verifier_model` (hint becomes optional when switching models), and the choice persists for subsequent attempts. Jobs can also carry a cheap-first `model_ladder` (`/job --model-ladder 'claude_code:haiku|claude_code:opus'`, `spawn_job(model_ladder=[...])`): when a rung exhausts its verifier attempts or the worker dies on a usage/rate limit, the next attempt escalates to the next model with a fresh budget. Each attempt records the model it ran on. Usage-limit worker deaths are classified as infrastructure failures (labeled, never counted as a quality failure).

### Notable fixes

- The web UI's event stream survives connection gaps: events carry sequence ids and replay on reconnect (laptop sleep/wake, network blips), a daemon restart is detected via a boot epoch and triggers a full refresh instead of stale rendering, slow clients are told to resync instead of silently losing events, zombie connections are detected and re-established, and the keystrip shows a "reconnecting" indicator while the stream is down. PWA resume reconnects the same stream so backgrounded-time events replay.
- `schedule_create`/`background_task` accept `notify=True`/bare-string values with a clear coercion or error instead of "'bool' object is not iterable".
- Provider-reported $0 costs (codex subscription models) record as 0.0 in usage.db instead of NULL, and compaction turns record their summarization cost; NULL now strictly means "no cost data".
- The sidebar search box searches all sessions server-side (title, topic, metadata; compacted sessions resolve to their live head) instead of only the ~100 loaded rows, and matches topic text client-side.
- A stranded "Working..." progress bubble self-heals: revisits reconcile against the event log, turn-end events finalize passively-watched sessions, and a watchdog settles quiet bubbles once the server reports not busy.

## 0.17.2 (2026-07-08)

### Features

- The model registry is now generated from models.dev instead of litellm: capabilities (reasoning support, effort levels, vision/audio, context/output limits, pricing) come from structured catalog data instead of model-name regexes. New models land in the registry, including Claude Fable 5, Sonnet 5, and the gpt-5.x line with their real effort vocabularies (`xhigh` included).
- The `claude_code` provider knows `claude-fable-5` and `claude-sonnet-5`, with `fable`/`fable-5`/`sonnet-5`/`sonnet-4-6` aliases. The bare `sonnet` alias now resolves to Sonnet 5; pin `sonnet-4-6` if you want the old model.
- The daemon always writes a rotating log file (`state_dir/daemon.log` unless `log_file` is set) and captures unhandled crash tracebacks from all threads, so a daemon crash is diagnosable after the fact.
- Daemon state (sessions, jobs, schedules, webhooks, terminals, push subscriptions, and API tokens) moved from per-store JSON files to a single `state_dir/daemon.db` (SQLite, write-through, owner-only permissions): every change is durable immediately instead of waiting for a flush, and a large session list no longer gets re-serialized wholesale on every save. Existing `*.json` state migrates automatically on first start and the old files are kept as backups. Like the history database, keep `state_dir` on local disk.
- `tsu daemon token create/list/revoke` now take `-c/--config` and resolve the daemon's `state_dir` from it, so tokens land where the daemon actually reads them - previously a daemon with a custom `state_dir` never saw CLI-created tokens without a manual file copy. Tokens created or revoked while the daemon runs take effect immediately.

### Notable fixes

- Jobs created without `--repo` are now verifiable: the worker, verifier, retries, and `file_exists:`/`cmd:` predicates all run anchored to the parent session's workspace, and the verifier reads the actual files before judging structural criteria - file-producing jobs no longer land in a false "stuck".
- The job verifier is more robust in general: noisy output around the verdict JSON (junk preambles, markdown fences, prose, duplicates) parses; a contradictory or partial verdict can't produce a false "done"; unparseable output surfaces an excerpt in the job error; retry workers get the original task restated instead of only "the verifier flagged X".
- Job worker and verifier sessions show as `<job-id> · worker` / `· verifier` in the sidebar instead of a bare session id, and opening a worker session now shows the code it ran, tool arguments, and tool output - including for turns interrupted by a stall or daemon restart.
- The native `anthropic` provider works with Opus 4.7+, Sonnet 5, and Fable 5: reasoning effort rides the models' adaptive-thinking surface (`xhigh` works) instead of the removed `budget_tokens` parameter, and unsupported sampling params are dropped instead of causing a 400.
- `fetch_text`, `fetch_json`, `http_request`, and `download_file` follow redirects by default (up to 5 hops, method and body preserved on 307/308) - 3xx responses were the most common tool failure. `follow_redirects=False` opts out, and `http_request` responses expose the final URL.
- `web_search` returns an empty list when a search finds nothing instead of raising and killing the surrounding code block.
- Pinned sessions always appear in the sidebar; the recent-sessions limit no longer evicts pins that have been idle for a while.
- The context-usage bar no longer disappears mid-session when a partial status event arrives.
- Progress messages from `send_message()` render as markdown, and pasting large text inline grows the input box immediately.
- The server is now the source of truth for whether a session is busy: session payloads and /status expose the authoritative turn-in-flight flag, the sidebar renders a running state from it even after a reconnect or mobile PWA resume with no fresh progress events, and a send that hits an in-flight turn restores your draft with a clear notice instead of a "Connection error".
- A turn that dies with the daemon (crash or restart mid-turn) is finalized at the next start: the conversation gets a visible "turn interrupted" note and a terminal event, so the session stops showing a stale "Waiting on LLM..." label forever and the interrupted turn's code and tool activity render in the conversation view instead of being hidden as still-streaming.

## 0.17.1 (2026-07-01)

### Notable fixes

- Sessions waiting on an `ask_user` reply get a distinct sidebar marker (peach dot + "waiting for your reply") so a blocked session is visible at a glance.
- The code expander no longer gets stuck showing "working..." after a turn that sent a progress message before asking a question.

## 0.17.0 (2026-06-30)

### Features

- tsugite is now a light core plus swappable plugins. `pip install tsugite-cli` gives you the CLI, built-in tools and providers. The `[daemon]`, `[web]`, `[sandbox]`, `[discord]`, and `[all]` extras add the rest.
- Conversation history is stored in one SQLite database instead of a JSONL file per session, searchable with `tsu history search` and shared across run, chat, workspace, and daemon mode. 
- Add api support for branching conversations, not exposed in the cli/ui yet.
- `tsu history export <id>` writes a conversation back to JSONL on demand.
- Agents tolerate uninstalled tools: a tool from a plugin you don't have is skipped with a warning instead of failing the run. Set `strict_tools: true` in frontmatter to require every listed tool.
- `tsu validate` reports tools an agent declares but that aren't installed (a warning, or an error under `strict_tools`).
- `sandbox.pass_env` exposes specific environment variable names to a sandboxed agent on top of the safe defaults.
- The package ships a `py.typed` marker so plugin authors' type checkers pick up tsugite's inline types.

### Breaking changes

- `pip install tsugite-cli` no longer includes the daemon, web search, or sandbox. Install `tsugite-cli[daemon]` (or `[all]`) to restore the previous full setup.
- The `--ui live` output mode was removed. The default plain mode and `--headless` remain.
- SQLite is the default history backend and `jsonl` is deprecated. Run `tsu history import` once to copy existing JSONL sessions into the database; the original `.jsonl` files are kept as a backup.
- Workspace sessions now live in the shared history database, not a per-workspace `session.jsonl`.
- Keep tsugite's data directory on local disk; SQLite history can corrupt on a network filesystem (NFS/SMB).

### Notable fixes

- Sandboxed agents no longer inherit the operator's environment: bwrap clears it and re-adds only a safe allowlist, so API keys and tokens in the environment can't be read by agent code. Secrets stay available through the masked `get_secret` tool.
- The tmux tools and the schedule remove/disable/cleanup tools are denied to sandboxed agents, and PTY sessions and spawned sub-agents/jobs inherit the parent's sandbox.
- Positional tool-call arguments work consistently across executors; passing both a positional and a keyword for the same argument now raises instead of silently overwriting.
- The web UI's Stop button now actually stops the agent: the run loop checks for a cancel between turns and before each tool/code step, so a stopped turn no longer keeps running (and writing history) in the background. A Stop mid-generation still finishes the current model call, then stops.
- The carried-forward conversation summary is one height-capped card that scrolls internally instead of pushing the conversation off-screen, and it no longer vanishes (along with the "load more" pagination button) when you re-open a session from the sidebar.
- The reasoning-effort selector re-derives its options when you change a session's model, so it shows the selected model's actual effort levels instead of the agent default's.
- `ask_user` / `ask_question` prompt buttons are styled to match the rest of the web UI instead of rendering as bare browser buttons.
- Stuck or errored Jobs can be dismissed (cancelled) from their tile, instead of being stranded with only "retry" or a misleading "mark done".
- A Job whose verifier returned its verdict JSON twice back-to-back is no longer wrongly marked stuck; the first verdict is used.
- Scheduled-run history cleanup no longer deletes the live conversation of an active schedule that reuses a fixed session id.
- A turn no longer re-stores the whole conversation, so history data is smaller.
