# v0.18

## (unreleased)

### Features

- Daemon adapter plugins are now a real extension point. A plugin registered under the `tsugite.adapters` entry point can mount its own HTTP routes under `/api/plugins/<name>` - authenticated with the daemon bearer token via `get_http_routes()`, or public (the plugin owns access control) via `get_public_http_routes()` - and register non-agent **job executors** via `get_job_executors()`. A job created with a registered `executor=` runs through that executor instead of an agent session while reusing all the existing job machinery: acceptance-criteria verification, the retry loop, stuck/mark-done/cancel, and the web tile. Route and executor collection is error-isolated, so a misbehaving plugin can't abort daemon startup. See `docs/plugin-adapters.md`.
- New `tsugite-cc-driver` plugin: run an interactive Claude Code (`claude`) session as a verified tsugite job (`executor="cc"`). A real `claude` process runs in a PTY driven by Claude Code HTTP `Stop` hooks - it keeps working until it emits a completion marker or spends its continue budget, then the attempt is graded by the normal job verifier (predicate + LLM acceptance criteria). A failed criterion is typed as a followup into the *same live session*, so context carries across attempts. From the new-job modal you pick the executor, and the job tile embeds the live terminal so you can watch the TUI and take over by typing at any time. Optional bubblewrap sandboxing (`sandbox: true`) gives filesystem isolation (job workspace + `~/.claude` read-write) with network left on. For headless, non-interactive automation use the `tsugite-claude-code` provider instead. See `plugins/tsugite-cc-driver/README.md`.
